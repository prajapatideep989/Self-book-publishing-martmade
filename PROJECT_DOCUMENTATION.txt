2. PROJECT INTRODUCTION
================================================================================

2.1 OVERVIEW
------------
The Self Book Publishing Mart is a comprehensive online platform designed to 
revolutionize the traditional book publishing industry by providing authors 
with direct access to readers. This web-based application eliminates the need 
for traditional publishing houses and intermediaries, empowering independent 
authors to publish, market, and sell their books directly to a global audience.

2.2 BACKGROUND
--------------
Traditional book publishing involves multiple gatekeepers, high upfront costs, 
and lengthy approval processes. Many talented authors struggle to get their 
work published due to these barriers. The digital age has created opportunities 
for self-publishing, but authors still face technical challenges in creating 
and managing their own platforms.

2.3 PROBLEM STATEMENT
---------------------
Current Challenges:
- High barrier to entry for new authors
- Expensive traditional publishing costs
- Limited control over pricing and distribution
- Technical complexity of self-publishing platforms
- Lack of quality control in existing platforms
- Unfair revenue sharing models

2.4 PROPOSED SOLUTION
--------------------
Alpha Book provides:
- Easy-to-use book upload and management system
- Secure payment processing through Razorpay
- Admin moderation for quality control
- Fair 90/10 revenue split (Author/Platform)
- User-friendly interface for buyers and sellers
- Comprehensive dashboard for tracking sales and earnings

7. DATABASE DESIGN
================================================================================

7.1 DATABASE NAME
-----------------
books_db

7.2 TABLES AND STRUCTURE
------------------------

TABLE 1: users
--------------
Purpose: Store user account information

Field Name      | Data Type      | Constraints              | Description
----------------|----------------|--------------------------|------------------
id              | INT(11)        | PRIMARY KEY, AUTO_INC    | Unique user ID
name            | VARCHAR(100)   | NOT NULL                 | User's full name
email           | VARCHAR(100)   | NOT NULL, UNIQUE         | Email address
phone           | VARCHAR(10)    | NOT NULL                 | Phone number
address         | TEXT           | NOT NULL                 | User address
password        | VARCHAR(255)   | NOT NULL                 | Hashed password
role            | ENUM           | DEFAULT 'user'           | 'admin' or 'user'

Indexes:
- Primary Key: id
- Unique Index: email

Sample Data:
- Admin: admin123@gmail.com / role: admin
- Users: deep1012@gmail.com, jay1012@gmail.com / role: user


TABLE 2: books
--------------
Purpose: Store book information uploaded by authors

Field Name        | Data Type      | Constraints           | Description
------------------|----------------|-----------------------|--------------------
id                | INT(11)        | PRIMARY KEY, AUTO_INC | Unique book ID
user_id           | INT(11)        | FOREIGN KEY, NOT NULL | Author's user ID
book_name         | VARCHAR(255)   | NOT NULL              | Book title
author_name       | VARCHAR(255)   | NOT NULL              | Author name
cover_image       | VARCHAR(255)   | NULL                  | Cover image path
description       | TEXT           | NOT NULL              | Book description
price             | DECIMAL(10,2)  | NOT NULL              | Book price
quantity          | INT(11)        | NOT NULL              | Available stock
publish_charge    | DECIMAL(10,2)  | DEFAULT 0.00          | Publishing fee
admin_commission  | DECIMAL(10,2)  | DEFAULT 0.00          | Platform commission
status            | ENUM           | DEFAULT 'pending'     | Status workflow
created_at        | TIMESTAMP      | DEFAULT CURRENT_TS    | Upload timestamp

Indexes:
- Primary Key: id
- Foreign Key: user_id REFERENCES users(id) ON DELETE CASCADE
- Index: user_id, status

Status Values:
- 'pending': Awaiting admin approval
- 'approved': Published on platform
- 'rejected': Denied by admin


TABLE 3: orders
---------------
Purpose: Store purchase transactions

Field Name         | Data Type      | Constraints           | Description
-------------------|----------------|-----------------------|--------------------
id                 | INT(11)        | PRIMARY KEY, AUTO_INC | Unique order ID
user_id            | INT(11)        | FOREIGN KEY, NOT NULL | Buyer's user ID
book_id            | INT(11)        | FOREIGN KEY, NOT NULL | Purchased book ID
quantity           | INT(11)        | NOT NULL              | Quantity purchased
total_price        | DECIMAL(10,2)  | NOT NULL              | Total cost
admin_earning      | DECIMAL(10,2)  | DEFAULT 0.00          | 10% platform fee
publisher_earning  | DECIMAL(10,2)  | DEFAULT 0.00          | 90% author earning
order_date         | TIMESTAMP      | DEFAULT CURRENT_TS    | Purchase timestamp

Indexes:
- Primary Key: id
- Foreign Key: user_id REFERENCES users(id) ON DELETE CASCADE
- Foreign Key: book_id REFERENCES books(id) ON DELETE CASCADE
- Index: user_id, book_id, order_date


TABLE 4: feedback
-----------------
Purpose: Store user feedback

Field Name    | Data Type      | Constraints           | Description
--------------|----------------|-----------------------|--------------------
id            | INT(11)        | PRIMARY KEY, AUTO_INC | Feedback ID
name          | VARCHAR(100)   | NOT NULL              | User name
email         | VARCHAR(100)   | NOT NULL              | User email
message       | TEXT           | NOT NULL              | Feedback message
submitted_at  | TIMESTAMP      | DEFAULT CURRENT_TS    | Submission time

7.3 ENTITY RELATIONSHIP DIAGRAM (ERD)
-------------------------------------

Relationships:

1. users (1) ----< (M) books
   - One user can upload multiple books
   - Each book belongs to one user (author)
   - Relationship: ONE-TO-MANY
   - Foreign Key: books.user_id --> users.id

2. users (1) ----< (M) orders
   - One user can place multiple orders
   - Each order belongs to one buyer
   - Relationship: ONE-TO-MANY
   - Foreign Key: orders.user_id --> users.id

3. books (1) ----< (M) orders
   - One book can have multiple orders
   - Each order contains one book (per record)
   - Relationship: ONE-TO-MANY
   - Foreign Key: orders.book_id --> books.id

Note: The system uses separate order records for multiple books in a single
purchase transaction.

7.4 DATABASE NORMALIZATION
--------------------------
The database is normalized to 3NF (Third Normal Form):

1NF (First Normal Form):
- All tables have atomic values
- Each column contains single values
- No repeating groups

2NF (Second Normal Form):
- All non-key attributes fully depend on primary key
- No partial dependencies

3NF (Third Normal Form):
- No transitive dependencies
- All non-key attributes depend only on primary key

7.5 DATA INTEGRITY CONSTRAINTS
------------------------------
- Primary Keys: Ensure uniqueness
- Foreign Keys: Maintain referential integrity
- NOT NULL: Mandatory fields
- UNIQUE: Prevent duplicates (email)
- DEFAULT: Set default values
- CASCADE: Auto-delete related records
- Enum: Restrict values (role, status)
- Data Type constraints: Validate input format
